# Migrate local-path to Longhorn

local-path stores data as a plain directory on one node's filesystem — no replication, no snapshots, no redundancy. Longhorn provides replicated block volumes that survive node failures, with snapshots and backup support.

Existing PVCs cannot be moved between storage classes in-place. Each PVC must be migrated by copying its data into a new Longhorn volume while the workload is stopped. This causes **downtime per workload** for the duration of the data copy.

**Estimated time:** 5–30 minutes per PVC depending on data size.

---

## How it works

```
Before                              After
──────────────────────              ──────────────────────
Pod → PVC → local-path PV          Pod → PVC (new) → Longhorn PV
             │                                         │
             ▼                                         ▼
        /var/lib/rancher/             Longhorn block volume
        k3s/storage/<pvc>/            replicated across nodes
        (single node, no HA)          (survives node failure)
```

The migration has four phases:

```
Phase 1 — Install Longhorn and switch the default StorageClass
Phase 2 — Migrate each PVC one at a time (workload down during copy)
Phase 3 — Verify all workloads are healthy on Longhorn
Phase 4 — Disable local-path (optional)
```

> **Downtime scope:** each workload is stopped for the duration of its data copy. Workloads not yet migrated keep running on local-path — you migrate one at a time. Plan for a maintenance window per workload or migrate all at once in a single window.

---

## Before you start

- [ ] Longhorn is installed: `bash kubernetes/k3s/helm/longhorn/install.sh`
- [ ] All Longhorn pods are Running: `kubectl get pods -n storage-controller`
- [ ] Longhorn UI is reachable at the configured ingress host
- [ ] You have `kubectl` access with cluster-admin privileges
- [ ] You have identified all local-path PVCs (Phase 1, step 1b)

---

## Phase 1 — Install Longhorn and switch the default StorageClass

### 1a — Install Longhorn

```bash
bash kubernetes/k3s/helm/longhorn/install.sh
```

Wait until all pods are Running before continuing:

```bash
kubectl get pods -n storage-controller -w
# All pods should show Running — this takes ~2 minutes
```

### 1b — Inventory all local-path PVCs

List every PVC still on local-path:

```bash
kubectl get pvc -A -o wide | grep local-path
```

For each namespace, find which workload owns each PVC:

```bash
kubectl get pods -n <namespace> -o json \
  | jq -r '.items[] | .metadata.name as $pod
    | .spec.volumes[]?
    | select(.persistentVolumeClaim != null)
    | [$pod, .persistentVolumeClaim.claimName] | @tsv'
```

Save the full list — you will work through it one PVC at a time in Phase 2.

### 1c — Switch the default StorageClass to Longhorn

```bash
bash kubernetes/k3s/helm/set-default-storageclass.sh
```

Verify:

```bash
kubectl get storageclass
# longhorn   (default)
# local-path
```

> From this point, any new PVC without an explicit `storageClassName` will use Longhorn automatically. Existing local-path PVCs are unchanged until Phase 2.

---

## Phase 2 — Migrate each PVC

Repeat the steps below for **every PVC** found in step 1b. Work through one PVC at a time — confirm the workload is healthy before moving to the next.

**Placeholders used throughout:**

| Placeholder   | Meaning                              |
| ------------- | ------------------------------------ |
| `<namespace>` | Namespace of the workload            |
| `<old-pvc>`   | Name of the existing local-path PVC  |
| `<new-pvc>`   | Name for the new Longhorn PVC        |
| `<size>`      | Same size as the old PVC (or larger) |
| `<workload>`  | Deployment or StatefulSet name       |

### 2a — Note the old PVC size

```bash
kubectl get pvc <old-pvc> -n <namespace> \
  -o jsonpath='{.spec.resources.requests.storage}'
```

### 2b — Scale the workload to zero

The workload must be stopped before copying — writing to the volume during copy will produce inconsistent data.

```bash
kubectl scale deployment <workload> --replicas=0 -n <namespace>

# Wait until all pods are fully terminated
kubectl wait pod -l app=<workload> -n <namespace> \
  --for=delete --timeout=120s
```

> **What happens to users during step 2b:**
> The workload is unavailable from this point until step 2g. For databases this means connection refused. Plan this window during off-peak hours or notify users in advance.

### 2c — Create the new Longhorn PVC

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: <new-pvc>
  namespace: <namespace>
spec:
  storageClassName: longhorn
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: <size>
EOF
```

Wait for it to bind — Longhorn provisions the PV automatically:

```bash
kubectl wait pvc <new-pvc> -n <namespace> \
  --for=jsonpath='{.status.phase}'=Bound --timeout=60s
```

### 2d — Copy data with a migration pod

This pod mounts both PVCs simultaneously and copies all data from old to new using `rsync`. The workload pod is not running — both volumes are safe to mount.

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: pvc-migrator
  namespace: <namespace>
spec:
  restartPolicy: Never
  containers:
    - name: migrator
      image: alpine
      command:
        - sh
        - -c
        - |
          apk add --no-cache rsync && \
          rsync -avh --progress /source/ /dest/ && \
          echo "Migration complete."
      volumeMounts:
        - name: source
          mountPath: /source
        - name: dest
          mountPath: /dest
  volumes:
    - name: source
      persistentVolumeClaim:
        claimName: <old-pvc>
    - name: dest
      persistentVolumeClaim:
        claimName: <new-pvc>
EOF
```

Follow progress:

```bash
kubectl logs -f pvc-migrator -n <namespace>
```

Wait for completion:

```bash
kubectl wait pod pvc-migrator -n <namespace> \
  --for=condition=Succeeded --timeout=30m
```

If the pod fails:

```bash
kubectl describe pod pvc-migrator -n <namespace>
kubectl logs pvc-migrator -n <namespace>
```

### 2e — Delete the migration pod

```bash
kubectl delete pod pvc-migrator -n <namespace>
```

### 2f — Update the workload to use the new PVC

**For a Deployment:**

```bash
kubectl patch deployment <workload> -n <namespace> \
  --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/volumes/0/persistentVolumeClaim/claimName","value":"<new-pvc>"}]'
```

Or edit manually:

```bash
kubectl edit deployment <workload> -n <namespace>
# Find the volumes section and change claimName to <new-pvc>
```

**For a StatefulSet** — see the StatefulSet section below.

### 2g — Scale the workload back up

```bash
kubectl scale deployment <workload> --replicas=1 -n <namespace>
kubectl rollout status deployment/<workload> -n <namespace>
```

### 2h — Verify the workload is healthy

```bash
kubectl get pods -n <namespace>
kubectl logs -n <namespace> -l app=<workload> --tail=50
```

Check that the application is reading its data correctly — connect to it, run a query, check a known record. Do not proceed to the next PVC until this one is confirmed healthy.

### 2i — Delete the old PVC

Only after confirming the workload is healthy:

```bash
kubectl delete pvc <old-pvc> -n <namespace>
```

> The underlying local-path PV and its directory on disk are deleted automatically (`reclaimPolicy: Delete`). Data on the old PVC is gone after this step — there is no recovery. If you are unsure, wait before running this command.

### 2j — Repeat for all remaining PVCs

Work through the full list from step 1b. Verify each workload before moving to the next.

---

## Phase 3 — Verify

### All PVCs are on Longhorn

```bash
kubectl get pvc -A -o wide
# No PVC should show local-path in the STORAGECLASS column
```

### All Longhorn volumes are healthy

```bash
kubectl get volumes.longhorn.io -n storage-controller
# All volumes should show state: attached or detached
# No volume should show state: degraded or faulted
```

### All workloads are Running

```bash
kubectl get pods -A | grep -v Running | grep -v Completed
# Should return nothing
```

---

## Phase 4 — Disable local-path (optional)

Once all PVCs are migrated, remove local-path as a StorageClass to prevent accidental use:

```bash
kubectl delete storageclass local-path
```

> k3s recreates the `local-path` StorageClass on every restart. To prevent this permanently, `DISABLE_LOCAL_STORAGE=true` must be set in `kubernetes/k3s/config.env` before provisioning — it passes `--disable local-storage` to the k3s server flags. On an already-running cluster, deleting the StorageClass object is sufficient but must be repeated after each k3s restart until the flag is set.

---

## Rollback

The old PVC is safe until step 2i. If anything goes wrong before that point:

```bash
# 1. Scale the workload to zero
kubectl scale deployment <workload> --replicas=0 -n <namespace>

# 2. Revert claimName back to the old PVC
kubectl patch deployment <workload> -n <namespace> \
  --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/volumes/0/persistentVolumeClaim/claimName","value":"<old-pvc>"}]'

# 3. Scale back up
kubectl scale deployment <workload> --replicas=1 -n <namespace>

# 4. Delete the failed new PVC and retry Phase 2 from step 2c
kubectl delete pvc <new-pvc> -n <namespace>
```

## StatefulSet migration

StatefulSets manage their own PVCs via `volumeClaimTemplates` and cannot be edited in-place to change the storage class. The procedure differs from Deployments.

**Example StatefulSet with 3 replicas named `kafka`:**

```
PVCs created by the StatefulSet:
  data-kafka-0   (local-path)
  data-kafka-1   (local-path)
  data-kafka-2   (local-path)
```

### Step 1 — Scale the StatefulSet to zero

```bash
kubectl scale statefulset <name> --replicas=0 -n <namespace>
kubectl wait pod -l app=<name> -n <namespace> --for=delete --timeout=120s
```

### Step 2 — Migrate each pod's PVC

For each PVC (`data-<name>-0`, `data-<name>-1`, etc.), follow Phase 2 steps 2c–2h above. Use a temporary name for the new PVC:

```
data-kafka-0  →  copy to  →  data-kafka-0-new
data-kafka-1  →  copy to  →  data-kafka-1-new
data-kafka-2  →  copy to  →  data-kafka-2-new
```

### Step 3 — Delete the StatefulSet without deleting PVCs

```bash
kubectl delete statefulset <name> -n <namespace> --cascade=orphan
```

> `--cascade=orphan` removes the StatefulSet controller object but leaves all pods and PVCs intact. Without this flag, Kubernetes would delete the pods and PVCs automatically.

### Step 4 — Delete the old local-path PVCs

```bash
kubectl delete pvc data-<name>-0 -n <namespace>
kubectl delete pvc data-<name>-1 -n <namespace>
kubectl delete pvc data-<name>-2 -n <namespace>
```

### Step 5 — Rename new PVCs to match the StatefulSet template pattern

StatefulSets expect PVCs named exactly `<template-name>-<pod-name>-<ordinal>`. Rename by creating new PVCs pointing to the same PVs and deleting the temporary ones:

```bash
# Get the PV name backing the temporary PVC
kubectl get pvc data-kafka-0-new -n <namespace> -o jsonpath='{.spec.volumeName}'

# Clear claimRef on the PV so it can be rebound
kubectl patch pv <pv-name> \
  --type=json \
  -p='[{"op":"remove","path":"/spec/claimRef"}]'

# Create the correctly named PVC pointing to the same PV
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-kafka-0
  namespace: <namespace>
spec:
  storageClassName: longhorn
  volumeName: <pv-name>
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: <size>
EOF

# Delete the temporary PVC (PV is already rebound — data is safe)
kubectl delete pvc data-kafka-0-new -n <namespace>
```

Repeat for each replica.

### Step 6 — Re-apply the StatefulSet manifest with Longhorn storage class

Update `volumeClaimTemplates` in your StatefulSet manifest:

```yaml
volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      storageClassName: longhorn # was local-path
      accessModes: [ReadWriteOnce]
      resources:
        requests:
          storage: <size>
```

```bash
kubectl apply -f <statefulset-manifest>.yaml -n <namespace>
```

### Step 7 — Verify pods attach to the correct PVCs

```bash
kubectl get pods -n <namespace>
kubectl get pvc -n <namespace>
kubectl logs -n <namespace> <pod-name> --tail=50
```

## Quick reference — full migration for one Deployment PVC

```bash
NS=<namespace>
OLD=<old-pvc>
NEW=<new-pvc>
SIZE=<size>
WORKLOAD=<workload>

# Scale down
kubectl scale deployment $WORKLOAD --replicas=0 -n $NS
kubectl wait pod -l app=$WORKLOAD -n $NS --for=delete --timeout=120s

# Create new Longhorn PVC
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: $NEW
  namespace: $NS
spec:
  storageClassName: longhorn
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: $SIZE
EOF

# Copy data
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: pvc-migrator
  namespace: $NS
spec:
  restartPolicy: Never
  containers:
    - name: migrator
      image: alpine
      command: [sh, -c, "apk add --no-cache rsync && rsync -avh /source/ /dest/ && echo done"]
      volumeMounts:
        - {name: source, mountPath: /source}
        - {name: dest,   mountPath: /dest}
  volumes:
    - {name: source, persistentVolumeClaim: {claimName: "$OLD"}}
    - {name: dest,   persistentVolumeClaim: {claimName: "$NEW"}}
EOF

kubectl wait pod pvc-migrator -n $NS --for=condition=Succeeded --timeout=30m
kubectl logs pvc-migrator -n $NS
kubectl delete pod pvc-migrator -n $NS

# Switch workload to new PVC and scale up
kubectl patch deployment $WORKLOAD -n $NS \
  --type=json \
  -p="[{\"op\":\"replace\",\"path\":\"/spec/template/spec/volumes/0/persistentVolumeClaim/claimName\",\"value\":\"$NEW\"}]"

kubectl scale deployment $WORKLOAD --replicas=1 -n $NS
kubectl rollout status deployment/$WORKLOAD -n $NS

# Verify healthy, then delete old PVC
kubectl delete pvc $OLD -n $NS
```

## Troubleshooting

| Symptom                                             | Cause                                    | Fix                                                                                                               |
| --------------------------------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| Migration pod stuck `Pending`                       | Longhorn cannot provision new PVC        | Check replica count vs storage nodes: `kubectl get pods -n storage-controller`                                    |
| `rsync` fails with permission denied                | Source volume has restricted permissions | Add `securityContext: runAsUser: 0` to the migration pod spec                                                     |
| Workload pod stuck `Pending` after switch           | PVC not yet bound                        | `kubectl describe pvc <new-pvc> -n <namespace>` — check Events                                                    |
| Workload starts but data is missing                 | `rsync` copied to wrong path             | Check source and dest mount paths in migration pod — trailing slash on `/source/` is required                     |
| local-path StorageClass reappears after k3s restart | `--disable local-storage` not set        | Set `DISABLE_LOCAL_STORAGE=true` in `kubernetes/k3s/config.env` and reprovision, or delete the StorageClass again |
| StatefulSet pods attach to new empty PVCs           | StatefulSet recreated before PVC rename  | Delete the empty PVCs, rename the migrated ones correctly, then recreate the StatefulSet                          |
