# Flannel to Cilium — Option B: Blue-Green Cluster

Build a second k3s cluster with Cilium running in parallel, migrate traffic at the load balancer level, then decommission the old cluster. Zero downtime — the old cluster stays fully operational until the new one is verified.

See `3.1_k3s_flannel_to_cillium.md` for background on the Flannel → Cilium migration. This doc covers the blue-green approach only.

**When to choose this over Option A (maintenance window):**

|            | Option A (3.2)    | Option B (this doc)               |
| ---------- | ----------------- | --------------------------------- |
| Downtime   | ~5 minutes        | None                              |
| Complexity | Low               | High                              |
| Extra cost | None              | Double infrastructure temporarily |
| Risk       | Rollback is fast  | Rollback means switching DNS back |
| Best for   | Off-peak clusters | SLA-critical production           |

---

## How it works

```
Old cluster (Flannel)          New cluster (Cilium)
─────────────────────          ─────────────────────
Production traffic      Step 1: provision new cluster
       │                Step 2: deploy all workloads
       │                Step 3: test and verify
       │                Step 4: switch load balancer / DNS
       └──────────────────────────────► New cluster now serves traffic
                        Step 5: decommission old cluster
```

The old cluster is never touched during the migration — rollback at any point before Step 4 is simply "do nothing". After Step 4 the rollback is a DNS/LB switch back to the old cluster IP.

---

## Step 1 — Provision new k3s cluster with Cilium

Provision fresh nodes and install k3s with Flannel disabled from the start — no migration needed on the new cluster.

```bash
# Set in kubernetes/k3s/config.env before provisioning
FIRST_SERVER_IP=<new-server-ip>
DISABLE_LOCAL_STORAGE=false

# In k3s config.yaml on each new server node — disable Flannel before install
flannel-backend: none
disable-network-policy: true

# Run full provisioning
ansible-playbook -i ansible/kubernetes-setup/inventory/hosts.ini \
  ansible/kubernetes-setup/playbook/site.yml

# Install Cilium immediately after cluster is up
helm repo add cilium https://helm.cilium.io
helm repo update cilium

helm install cilium cilium/cilium \
  --version 1.17.0 \
  -f kubernetes/essential/cilium/values.yaml \
  -n kube-system --wait --timeout 5m
```

Verify the new cluster is healthy:

```bash
# Point kubectl to the new cluster
export KUBECONFIG=~/.kube/k3s-new.yaml

kubectl get nodes          # all Ready
kubectl get pods -A        # Cilium pods Running in kube-system
```

## Step 2 — Deploy all workloads to the new cluster

### 2a — Install cluster essentials

```bash
bash kubernetes/k3s/helm/longhorn/install.sh
bash kubernetes/k3s/helm/ingress-nginx/install.sh
```

### 2b — Apply all workload manifests

```bash
kubectl apply -f manifests/
```

### 2c — Migrate persistent data

For each PVC on the old cluster, copy its data to the new cluster:

```bash
# On old cluster — create a temporary NodePort service to expose the data
# Or use kubectl cp to pull data out:
kubectl cp <namespace>/<pod>:/data ./backup-data

# On new cluster — copy data into the new PVC via a migration pod
# Follow the same procedure as 6.1_k3s_local_path_to_longhorn.md
```

> Data migration is the most time-consuming part of a blue-green migration. For databases, use database-native tools (see `6.3`) rather than filesystem copy to ensure consistency.

### 2d — Configure the new cluster to point at the same external dependencies

Update any references to external services (databases, message queues, object stores) in the new cluster's secrets and configmaps to match the old cluster's configuration.

---

## Step 3 — Verify new cluster is healthy

Test the new cluster thoroughly before switching any traffic. The old cluster is still live — users are unaffected by anything that goes wrong here.

```bash
# All pods running
kubectl get pods -A | grep -v Running | grep -v Completed

# Test application endpoints directly against the new ingress IP
# (not via DNS — DNS still points to the old cluster)
curl http://<new-ingress-ip>/healthz
curl -H "Host: myapp.example.com" http://<new-ingress-ip>/api/status

# Run your full smoke test suite against the new cluster IP
```

> Use the new cluster's ingress IP directly with a `Host` header override, or add a temporary `/etc/hosts` entry on your workstation. Do not change public DNS yet.

```bash
# Temporary /etc/hosts entry for testing
echo "<new-ingress-ip>  myapp.example.com" >> /etc/hosts
# Test thoroughly, then remove the entry before Step 4
```

---

## Step 4 — Switch traffic to new cluster

### 4a — Lower DNS TTL (day before)

Lower the TTL on your DNS records to 60 seconds so that the switch propagates quickly when you make it:

```
myapp.example.com  A  <old-ingress-ip>  TTL: 60
```

> If TTL is still high (e.g. 3600s) when you switch, clients will continue hitting the old cluster for up to an hour after the change.

### 4b — Switch DNS or load balancer

```bash
# Example: update DNS A record
# Before: myapp.example.com  →  <old-ingress-ip>
# After:  myapp.example.com  →  <new-ingress-ip>
```

For an L4 load balancer in front of both clusters, update the backend pool to point to the new ingress node instead of the old one.

### 4c — Monitor for 15–30 minutes

```bash
# Watch error rates and latency on the new cluster
# Check application logs for unexpected errors
kubectl logs -n <namespace> -l app=<workload> -f --tail=100

# Confirm old cluster is receiving no new traffic
# (check ingress access logs on old cluster — should go quiet)
```

Do not decommission the old cluster until you are confident the new one is stable.

---

## Step 5 — Decommission old cluster

Only after confirming the new cluster is stable and the old cluster is receiving no traffic:

```bash
ansible-playbook -i ansible/kubernetes-setup/inventory/hosts-old.ini \
  ansible/kubernetes-setup/playbook/clean.rhel.yml
```

---

## Rollback

**Before Step 4 (DNS switch):** do nothing — the old cluster is still live and unchanged. Delete the new cluster and retry.

**After Step 4 (DNS switched):** point DNS back to the old cluster IP. With TTL at 60s this takes effect within 60 seconds.

```
DNS switch back:
  myapp.example.com  →  <old-ingress-ip>
Takes effect in: 60s (if TTL was lowered in Step 4a)
```

The old cluster remains fully operational until Step 5. Keep it running for at least 24 hours after the DNS switch before decommissioning — long enough to catch any issues that appear under real traffic.
