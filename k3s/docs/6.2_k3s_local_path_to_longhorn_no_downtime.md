# Migrate local-path to Longhorn — No Downtime

The standard migration in `6.1_k3s_local_path_to_longhorn.md` stops the workload during the data copy. This doc covers the no-downtime approach — the workload keeps serving traffic while data is copied in the background, and traffic is cut over with a rolling restart at the end.

**The trade-off:**

|                            | With downtime (6.1)     | No downtime (this doc)              |
| -------------------------- | ----------------------- | ----------------------------------- |
| Workload availability      | Down during copy        | Always up                           |
| Complexity                 | Low                     | Medium                              |
| Risk of data inconsistency | None — workload stopped | Low — mitigated by copy-then-verify |
| Best for                   | Non-critical workloads  | Production stateful apps            |

> **No-downtime does not mean zero risk.** During the copy window, the workload is still writing to the old PVC. `rsync` copies a point-in-time snapshot — any writes that land after the copy finishes are not in the new volume. The final step is a brief rolling restart (seconds, not minutes) to switch the pod to the new PVC. For stateless-friendly apps this is invisible; for strict databases, use the database-native approach in `6.3`.

---

## How it works

```
Phase 1 — Copy data while workload is running (live copy)
              │
              ▼
          old PVC ──rsync──► new Longhorn PVC
          (writes continue)   (catching up)

Phase 2 — Rolling restart to switch the pod to the new PVC
              │
              ▼
          Pod restarts with new PVC mounted
          Old PVC unmounted — no more writes go to it
          (restart takes ~10–30 seconds)

Phase 3 — Verify, then delete old PVC
```

> **Why rsync works here:** rsync is incremental — run it once to bulk-copy the data, then run it again just before the rolling restart to sync only the delta (new writes since the first copy). The delta copy is fast, so the final switch window is tiny.

---

## Before you start

- [ ] Longhorn is installed and healthy: `kubectl get pods -n storage-controller`
- [ ] Default StorageClass switched to Longhorn: `bash kubernetes/k3s/helm/set-default-storageclass.sh`
- [ ] Workload has at least 2 replicas on different nodes (for zero-downtime rolling restart)
- [ ] You have `kubectl` access with cluster-admin privileges

> **Single-replica workloads will have a brief restart gap** (~10–30 seconds) during the rolling restart in Phase 2. This is unavoidable without 2+ replicas. If true zero downtime is required, scale to 2 replicas before starting:
>
> ```bash
> kubectl scale deployment <workload> --replicas=2 -n <namespace>
> ```

## Phase 1 — Live copy

### 1a — Note the old PVC size

```bash
kubectl get pvc <old-pvc> -n <namespace> \
  -o jsonpath='{.spec.resources.requests.storage}'
```

### 1b — Create the new Longhorn PVC

The workload keeps running on the old PVC while this is created:

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: <new-pvc>
  namespace: <namespace>
spec:
  storageClassName: longhorn
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: <size>
EOF

kubectl wait pvc <new-pvc> -n <namespace> \
  --for=jsonpath='{.status.phase}'=Bound --timeout=60s
```

### 1c — Run the first rsync (bulk copy)

This pod mounts the old PVC read-only and the new PVC read-write. It copies all current data. The workload pod continues writing to the old PVC in parallel — this is safe because we mount the source as read-only and run a second sync before switching.

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: pvc-migrator
  namespace: <namespace>
spec:
  restartPolicy: Never
  containers:
    - name: migrator
      image: alpine
      command:
        - sh
        - -c
        - |
          apk add --no-cache rsync && \
          rsync -avh --progress /source/ /dest/ && \
          echo "First sync complete."
      volumeMounts:
        - name: source
          mountPath: /source
          readOnly: true
        - name: dest
          mountPath: /dest
  volumes:
    - name: source
      persistentVolumeClaim:
        claimName: <old-pvc>
        readOnly: true
    - name: dest
      persistentVolumeClaim:
        claimName: <new-pvc>
EOF
```

Follow progress:

```bash
kubectl logs -f pvc-migrator -n <namespace>
```

Wait for completion:

```bash
kubectl wait pod pvc-migrator -n <namespace> \
  --for=condition=Succeeded --timeout=60m

kubectl delete pod pvc-migrator -n <namespace>
```

> The first sync may take 10–30 minutes for large volumes. The workload is fully operational throughout — users are unaffected.

## Phase 2 — Delta sync and rolling restart

The bulk copy is done. Now sync the delta (writes that landed during Phase 1), then do the rolling restart while the delta is still small.

> **Run Phase 2 steps without pausing.** The delta grows every second the workload writes. Run 2a → 2b → 2c back to back.

### 2a — Run the delta rsync

Mount the old PVC read-only again and sync only what changed since Phase 1:

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: pvc-migrator-delta
  namespace: <namespace>
spec:
  restartPolicy: Never
  containers:
    - name: migrator
      image: alpine
      command:
        - sh
        - -c
        - |
          apk add --no-cache rsync && \
          rsync -avh --checksum --progress /source/ /dest/ && \
          echo "Delta sync complete."
      volumeMounts:
        - name: source
          mountPath: /source
          readOnly: true
        - name: dest
          mountPath: /dest
  volumes:
    - name: source
      persistentVolumeClaim:
        claimName: <old-pvc>
        readOnly: true
    - name: dest
      persistentVolumeClaim:
        claimName: <new-pvc>
EOF

kubectl wait pod pvc-migrator-delta -n <namespace> \
  --for=condition=Succeeded --timeout=10m

kubectl delete pod pvc-migrator-delta -n <namespace>
```

> `--checksum` makes rsync compare file contents rather than just timestamps — catches any writes that landed on already-existing files during Phase 1.

### 2b — Patch the workload to use the new PVC

```bash
kubectl patch deployment <workload> -n <namespace> \
  --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/volumes/0/persistentVolumeClaim/claimName","value":"<new-pvc>"}]'
```

### 2c — Trigger a rolling restart

The patch in 2b updates the pod template, which triggers a rolling restart automatically. With 2+ replicas, Kubernetes replaces pods one at a time — at least one replica stays up throughout.

```bash
kubectl rollout status deployment/<workload> -n <namespace>
# Watch pods replace one at a time
```

> **What happens during the rolling restart:**
>
> ```
> Replica 1: still running on old PVC
> Replica 2: terminated → new pod starts → mounts new PVC → passes health check
> Replica 1: terminated → new pod starts → mounts new PVC → passes health check
>
> All traffic now goes to pods on new PVC.
> Old PVC has no pods — no more writes.
> ```
>
> With 2+ replicas on different nodes and a working readiness probe, users see no interruption.

## Phase 3 — Verify and clean up

### Verify the workload is reading from the new PVC

```bash
kubectl get pods -n <namespace> -o wide
# All pods should be Running on the new PVC

kubectl logs -n <namespace> -l app=<workload> --tail=50
# Application logs should show normal operation
```

### Confirm no pod is still mounted to the old PVC

```bash
kubectl get pods -n <namespace> -o json \
  | jq -r '.items[].spec.volumes[] | select(.persistentVolumeClaim.claimName == "<old-pvc>") | .persistentVolumeClaim.claimName'
# Should return nothing — no pod references the old PVC
```

### Delete the old PVC

Only after confirming no pod uses it:

```bash
kubectl delete pvc <old-pvc> -n <namespace>
```

---

## Rollback

At any point before deleting the old PVC in Phase 3:

```bash
# Revert claimName back to old PVC
kubectl patch deployment <workload> -n <namespace> \
  --type=json \
  -p='[{"op":"replace","path":"/spec/template/spec/volumes/0/persistentVolumeClaim/claimName","value":"<old-pvc>"}]'

# Rolling restart back to old PVC
kubectl rollout status deployment/<workload> -n <namespace>

# Delete the new PVC
kubectl delete pvc <new-pvc> -n <namespace>
```

Rollback is also a rolling restart — same near-zero downtime as the cutover.

## Limitations

**This approach does not work for:**

| Workload type                                                    | Why                                                                                              | Alternative                                                  |
| ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------ |
| Databases with strict consistency (PostgreSQL, MySQL)            | rsync copies files while the DB engine is writing — page files may be mid-write and inconsistent | Use `6.3` — database-native backup/restore                   |
| Single-writer volumes where the workload holds an exclusive lock | The source PVC cannot be mounted read-only while the workload holds an exclusive lock on it      | Stop the workload briefly (6.1) or use app-level replication |
| Very high write rate workloads                                   | Delta between Phase 1 and Phase 2 grows too fast — delta sync takes too long                     | Pre-size the maintenance window or use app-level replication |

> For PostgreSQL and other databases that hold open file handles and write-ahead logs, rsync produces a physically inconsistent copy — the data files may be internally inconsistent even if the filesystem copy completes without error. Always use `pg_basebackup` or a database snapshot for Postgres. See `6.3_k3s_local_path_to_longhorn_db.md`.

## Choosing between 6.1 and 6.2

|                           | Use 6.1 (with downtime) | Use 6.2 (no downtime)             |
| ------------------------- | ----------------------- | --------------------------------- |
| Workload type             | Any                     | Stateless or file-based apps only |
| Replica count             | Any                     | 2+ recommended                    |
| Database                  | Safe                    | Not safe — use 6.3                |
| Maintenance window needed | Yes                     | No                                |
| Complexity                | Low                     | Medium                            |
